
# Centralized Logging and Troubleshooting Overview

The OCI HPC Stack 3.0 generates logs at multiple layers of the system, covering infrastructure provisioning, cluster management, node initialization, and job execution. Terraform and Resource Manager logs capture deployment-time infrastructure events, while mgmt service logs provide visibility into cluster orchestration, node registration, and Slurm topology management. Node-level configuration logs record cloud-init and Ansible-based setup for compute and login nodes, and Slurm job logs capture application-level output and runtime behavior. Together, these logs form a comprehensive troubleshooting framework that enables administrators and users to quickly identify and resolve issues across the full lifecycle of the HPC cluster.

### Post-Deployment Initial Configuration Logs

After the OCI HPC Stack 3.0 cluster is successfully deployed, each node performs an initial configuration process that installs packages, configures system services, and applies cluster-specific settings. The logs generated during this phase are critical for validating a successful deployment and for troubleshooting any post-deployment issues.

On all cluster nodes (head node, compute nodes, login nodes, and GPU nodes), the initial configuration logs are available at: `/config/logs/initial_configure.log`

To verify that the initial configuration completed successfully:

```bash
sudo tail -n 50 /config/logs/initial_configure.log
```

For a quick scan of errors:

```bash
sudo grep -iE "error|failed|fatal" /config/logs/initial_configure.log
```

### Terraform Deployment Logs (Resource Manager)

The Terraform execution logs for the HPC Stack deployment are available through Oracle Resource Manager in the OCI Management Console. Access the Terraform logs by logging in to the OCI Management Console, navigating to Developer Services → Resource Manager, selecting the relevant Stack, and opening the Latest Job.

Also to validate the successful deployment of the cluster and the GPU nodes, run `mgmt nodes list` from any node in the cluster.

### Cluster Management (mgmt) logs

The Cluster Management (mgmt) service is a core component of the OCI HPC Stack 3.0 and is responsible for orchestrating cluster-level operations such as registering nodes in the internal database, updating and maintaining the Slurm topology, managing images and configurations, and coordinating network and fabric-related settings. All logs generated by the mgmt service are stored in `/config/logs/mgmt.log` and should be the first point of inspection when troubleshooting issues related to node onboarding, Slurm inconsistencies, or service-level failures. The mgmt CLI exposes multiple administrative command groups, including cluster and node lifecycle management (clusters, nodes, login), configuration and image management (configurations, images), database operations (database), network and fabric visibility (network, fabrics), service control (services), and advisory insights (recommendations), making it a central tool for diagnosing and managing the overall health of the HPC cluster.

### Compute Node Logs

Each node in the cluster performs self-configuration during boot by executing a cloud-init script, with a copy available at `/config/bin/cloud-init.sh`. Based on the assigned role, the node then runs a role-specific initialization script: compute nodes execute `/config/bin/compute.sh`, which in turn runs the compute.yml Ansible playbook, while login nodes execute `/config/bin/login.sh`, which runs the `login.yml` Ansible playbook. Logs generated during this configuration process are stored on each node at `/config/logs/${hostname}.log` and should be reviewed when troubleshooting node-level initialization or role-specific configuration issues.

### Slurm (Job) Logs

Slurm job output and error logs are generated on a per-job basis and are written to the directory from which the job is submitted, unless explicitly overridden in the job submission script. By default, these logs follow the naming convention `slurm-<jobid>.out` and contain the combined standard output and standard error produced during job execution. These files are essential for troubleshooting job failures, application errors, resource allocation issues, and runtime warnings, and users should review them first when a Slurm job terminates unexpectedly or produces incorrect results.

## Health Checks

### Passive Health Checks

Passive health checks allow you to verify the status of critical hardware components, such as RDMA interfaces and GPUs, without consuming significant compute resources. These checks can be safely run on busy compute nodes and are useful for routine monitoring or pre-job validation. To perform a passive health check, run the following command on any compute node:

```bash
sudo python3 /opt/oci-hpc/healthchecks/check_gpu_setup.py
```
This script inspects the GPU drivers, CUDA environment, and RDMA interfaces, reporting any configuration issues or hardware problems that could impact cluster performance.

### Active Health Checks

Active health checks are designed to validate the functional readiness of compute nodes by running lightweight test workloads that exercise GPUs, RDMA, and related system components. To run the active health checks directly on a node, execute the following command:

```bash
sudo python3 /opt/oci-hpc/healthchecks/active_healthcheck.py
```

Alternatively, the active health checks can be submitted from the Slurm controller and targeted to a specific node by replacing hostname with the desired node name:

```bash
sbatch -N 1 -w hostname /opt/oci-hpc/healthchecks/active_HC.sbatch
```

The results of the above command are written to `/var/log/healthchecks/active_HealthCheck` on the target node. These checks are also executed automatically once per day on idle nodes using a low-priority Slurm partition (compute_healthcheck) to avoid impacting production workloads. While the checks are running, detailed output is written to `/var/log/healthchecks/latest_active_healthcheck.log` on the target node, and the results can also be viewed from the controller using the `mgmt nodes get <hostname>` command.

### Multi-Node Active Health Checks

Multi-node active health checks are used to validate inter-node functionality, including multi-node GPU communication and RDMA fabric behavior. To run these checks manually, submit the following Slurm job from the controller and replace hostnames with the target node names using the -w option:

```bash
sbatch -N 2 -w hostname1,hostname2 /opt/oci-hpc/healthchecks/multi_node_active_HC.sbatch
```

The results of the above command are written to `/var/log/healthchecks/multi_node_active_HealthCheck` on the target nodes. These multi-node checks are also executed automatically once per day on idle nodes through the low-priority compute_healthcheck partition to ensure continuous validation without impacting production workloads. During execution, detailed output is written to `/var/log/healthchecks/latest_multi_node_active_healthcheck.log` on the participating nodes, and the aggregated results can be reviewed from the controller using the `mgmt nodes get <hostname>` command.

#### NCCL Output and Logs

The script classifies nodes into the following categories based on test results:

- Good Nodes: Nodes that meet or exceed the expected bandwidth threshold
- Bad Nodes: Nodes that fall below the bandwidth threshold
- Unreachable Nodes: Nodes that cannot be accessed via SSH
- Timeout Nodes: Nodes that fail to respond within the test duration


#### Automated Execution
The `/opt/oci-hpc/healthchecks/ncclscout.py` script is executed automatically as part of the Slurm job prolog and is also run periodically on idle or drained nodes every five minutes, as configured by the `HealthCheckInterval` parameter in `/etc/slurm/slurm.conf`. This ensures continuous validation of GPU and NCCL health across the cluster without impacting active workloads.


### Slurm Reservation

During the initial provisioning and validation phase, a Slurm reservation named InitialValidation is created to prevent newly added nodes from being scheduled for user workloads before health checks and configuration validation are completed. This reservation ensures that nodes remain isolated while initial setup, GPU checks, and connectivity tests are performed.

You can inspect the reservation details using the following command:

```bash
scontrol show reservation InitialValidation
```

If any issues are encountered during validation, individual nodes can be removed from the reservation, or the reservation itself can be deleted, to allow corrective actions or reconfiguration. To avoid unnecessarily keeping resources unavailable, nodes that complete the active health checks but do not pass or complete the multi-node health checks are automatically removed from the reservation after 10 minutes. This mechanism ensures that single nodes are not held in an unavailable state for an extended period while still maintaining cluster stability during initial validation.
